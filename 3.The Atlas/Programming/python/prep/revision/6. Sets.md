# 6. Sets

## First Question

Why did Python create Sets?

Imagine you have a list.

```python
nums = [10,20,30,20,10,40]
```

Now someone asks

> Is **20** present?

Python must search.

```
10 ❌

20 ✅
```

Worst case

```
O(n)
```

Now imagine

10 million numbers.

Searching becomes expensive.

Python says

> Let's build a data structure optimized ONLY for membership checking.

That is a Set.

---

# Mental Model

A Set is

```
A bag of unique keys
```

Notice

Not values.

Not key-value pairs.

Just keys.

---

# Internal Structure

```python
nums = {10,20,30}
```

Internally

```
Hash Table

hash(10)

↓

Bucket 5

10



hash(20)

↓

Bucket 1

20



hash(30)

↓

Bucket 8

30
```

Notice

Python doesn't store

```
10

20

30
```

in order.

It stores them according to

```
hash(value)
```

This is why sets are **unordered**.

---


Instead

```

↓

Bucket 1

↓

```

Average

```
O(1)
```

This is the entire reason Sets exist.

## The Deep Intuition

Forget the syntax for a moment.

A **list** is like a notebook:

```
Page 1 → 10
Page 2 → 20
Page 3 → 30
```

You care about **position**. That's why you have indexing.

A **set** is like a security gate with a guest database:

```
Is Alice allowed?
```

The guard doesn't care if Alice was the "3rd person added." He only cares whether Alice is **in the system**. He looks up her name using a hash and answers almost instantly.

That's why sets have:

- ❌ No indexing
    
- ❌ No guaranteed order
    
- ✅ Very fast membership tests
    
- ✅ Automatic duplicate removal
    

---

### Connection to C++

|C++|Python|
|---|---|
|`unordered_set`|`set`|
|`unordered_map`|`dict`|
|`set` (tree-based)|No direct equivalent|

So when you think of Python's `set`, think **`unordered_set`** more than `std::set`, because both are hash-table based and optimized for fast lookup.
