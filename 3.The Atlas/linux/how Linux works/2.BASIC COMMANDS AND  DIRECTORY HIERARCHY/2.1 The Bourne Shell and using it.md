# 2.1 The Bourne Shell (`/bin/sh`)

## What is a Shell?

When you use Linux, you don't talk directly to the kernel.

The kernel understands only low-level requests through **system calls**. Humans cannot conveniently interact with the kernel directly.

That's where the **shell** comes in.

A shell is a program that:

- Accepts commands from users
    
- Interprets those commands
    
- Starts programs
    
- Connects programs together
    
- Provides a scripting environment
    

Think of the shell as a **translator between humans and Linux**.

```text
User
  ↓
Shell
  ↓
Kernel
  ↓
Hardware
```

---

## Why Do We Need a Shell?

Imagine wanting to list files.

Without a shell:

```text
You would need to manually invoke system calls.
```

With a shell:

```bash
ls
```

Much easier.

The shell provides a human-friendly way to interact with Linux.

---

## The Shell is Just Another Program

One of the most important Linux ideas:

> The shell is not the operating system.

The shell itself runs as a **user-space process**.

Just like:

- Chrome
    
- Firefox
    
- VS Code
    
- Python
    

Bash is simply another program running on Linux.

```text
User Space
│
├── bash
├── firefox
├── nginx
├── python
└── vscode

Kernel Space
│
└── Linux Kernel
```

---

## Shell as a Programming Environment

The shell does more than execute commands.

It can:

- Store variables
    
- Execute loops
    
- Make decisions
    
- Create functions
    
- Run scripts
    

Example:

```bash
for i in 1 2 3
do
    echo $i
done
```

Output:

```text
1
2
3
```

This is why Linux administrators automate tasks using shell scripts.

---

# Shell Scripts

A shell script is simply a text file containing shell commands.

Example:

```bash
#!/bin/bash

echo "Starting Backup"
date
cp file1 backup/
```

Instead of typing commands repeatedly:

```bash
echo "Starting Backup"
date
cp file1 backup/
```

Store them once in a script and execute:

```bash
./backup.sh
```

---

## Why Shell Scripts Matter

System administrators often need to:

- Create users
    
- Backup files
    
- Monitor servers
    
- Restart services
    

Doing these manually is tedious.

Shell scripts automate everything.

The original Unix shell was the **Bourne Shell**.

Created at:

```text
Bell Labs
```

by:

```text
Stephen Bourne
```

It became the standard shell for Unix systems.

Path:

```bash
/bin/sh
```

## Why is Bourne Shell Important?

Almost every modern shell is based on ideas introduced by the Bourne Shell.

Examples:

- Bash
    
- Dash
    
- Korn Shell (ksh)
    
- Z Shell (zsh)

## Relationship Between `/bin/sh` and Bash

On most Linux systems:



# Unix Input and Output

Every Linux program communicates using streams.

Think of streams as pipes carrying data.

```text
Input → Program → Output
```

---

# Standard Input (stdin)

Default input stream.

Abbreviation:

```text
stdin
```

Usually connected to:

```text
Keyboard
```

---

Example:

```bash
cat
```

No file provided.

Now type:

```text
Hello
```

Output:

```text
Hello
```

Because cat reads from stdin.

---

# Standard Output (stdout)

Default output stream.

Abbreviation:

```text

```

Usually connected to:

```text
Terminal Screen
```

Example:

```bash

```


# How Everything Connects Together

When you type:

```bash
cat file.txt
```

Internally:

```text
Keyboard
   ↓
Shell (bash)
   ↓
fork()
   ↓
exec(cat)
   ↓
cat Process
   ↓
open(file.txt)
read(file.txt)
   ↓
stdout
   ↓
Terminal
```

Notice the connection:

```text
Shell
   ↓
Starts Processes
   ↓
Processes Use System Calls
   ↓
Kernel Performs Work
   ↓
Results Return To User
```

This is one of the most important ideas in Linux and will keep appearing throughout the rest of the book.
